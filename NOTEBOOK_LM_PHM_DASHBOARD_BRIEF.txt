EXECUTIVE SUMMARY (USE AS SLIDE 2 — AFTER TITLE PAGE)
-----------------------------------------------------
We built a browser-based dashboard for NASA’s Commercial Modular Aero-Propulsion System Simulation (C-MAPSS) turbofan engine degradation dataset. The application helps engineers and managers understand Remaining Useful Life (RUL) prediction in a fleet-health context: what is measured, how training vs test splits differ, which exploratory plots matter for feature selection, and how modern deep-learning choices (1D CNN, Transformer, PINN-style physics priors) compare for aerospace monitoring. The stack is Python, Streamlit, Plotly, and Pandas; the UI supports Dark/Light appearance, downloadable NASA readme text, and an auto-generated PDF analysis brief per selected dataset.


PROBLEM DOMAIN: WHY TURBofan PHM MATTERS
----------------------------------------
- Modern aircraft engines are safety-critical. Operators need estimates of how many flight cycles remain before maintenance or failure risk becomes unacceptable.
- Prognostics and Health Management (PHM) uses sensor time series (temperatures, pressures, speeds, flows) to infer degradation state and predict RUL.
- NASA C-MAPSS provides a standard benchmark: simulated multi-engine fleets with controlled fault modes and operating conditions, enabling reproducible research without exposing proprietary airline data.


DATASET: NASA C-MAPSS (FD001–FD004)
-----------------------------------
- Each record is one “snapshot” per engine per flight cycle: unit ID, cycle index, three operational settings, and twenty-one sensor channels (26 numeric fields total in the standard layout).
- Training runs are “run-to-failure” in simulation; test runs are censored (the engine stops before failure in the log), and a separate RUL_FD00x.txt file gives the true remaining cycles for each test engine at the end of its series.
- Four scenarios (metadata summarized in-app from NASA readme):
  FD001: single operating condition (sea level); fault focused on high-pressure compressor (HPC) degradation; 100 train / 100 test engines (typical counts from readme).
  FD002: six mixed flight regimes; HPC-only fault; larger fleet counts (on the order of hundreds of engines per split).
  FD003: sea level; combined HPC and fan degradation; 100/100 style counts.
  FD004: mixed regimes; HPC + fan; larger fleet counts.
- Constant (near-zero variance) sensors are common in some splits; the dashboard detects and lists them because they are often dropped in published papers.


THREE ENGINEERING PILLARS (FRAMING FOR MANAGEMENT)
--------------------------------------------------
The app narrative is organized around three pillars for briefing decision-makers:

1) Physical system insight  
   What the turbofan subsystem context is (e.g., HPC, combustor, turbine proxies), what 26 channels represent, and how degradation may appear as trends in temperature-like vs pressure-margin-like traces over cycles.

2) Model trade-offs  
   When purely data-driven sequence models (1D CNN sliding windows, Transformer attention over time) are appropriate versus when embedding structure via physics-informed neural networks (PINNs) or hybrid stacks (e.g., GNN + Transformer + PINN) is justified—especially under data scarcity or assurance needs in aerospace.

3) Compute and data reality  
   Data quality concerns: missing-not-at-random (MNAR) sensor dropouts during transients, calibration drift vs real degradation, class imbalance (many nominal cycles vs rare fault regimes), and deployment cost drivers (GEMM-heavy layers, attention KV cache memory, quantization for edge inference).

MAIN TAB STRUCTURE (SIX TABS)
----------------------------

Tab 1 — Fleet overview  
- Interactive multi-engine time-series view: two stacked subplots for user-selected primary and secondary sensor channels vs cycle.  
- Multiselect engine units; default picks a small subset for speed.  
- Short explanatory text ties sensor choices to literature-style proxies (e.g., LPT outlet, Ps30 pressure).  
- Plotly theme applied for consistent axes and backgrounds.

Tab 2 — Exploratory analysis (EDA)  
- Checkbox to include training split in EDA tables/plots when both train and test exist.  
- Metrics: row count, engine count, varying vs constant sensors, train rows, RUL label count.  
- Expandable list of constant sensors.  
- Plots (all driven from real loaded columns): Pearson correlation heatmap among varying sensors (row-sampled for large tables); bar chart of absolute correlation with cycle index (trend strength); last-cycle per engine and run-length histograms; 2D settings scatter and 3D settings cloud; per-sensor standard deviation; optional train vs test last-cycle histogram overlay on one sensor; RUL overview if labels exist; fleet-mean normalized-life trajectories for up to four sensors; PCA on last-cycle standardized vectors; optional pair-sensor scatter at last snapshot.  
- Purpose: support feature discovery, regime understanding, and discussion of label alignment with sensor behavior.

Tab 3 — Data audit report  
- Qualitative “Session 4 style” engineering dimensions: Completeness, Consistency, Accuracy, Timeliness, Relevance—with illustrative radar chart and a companion table (scores and discussion hooks).  
- Text explains MNAR risk, calibration drift, and rare-fault imbalance.  
- Intended as a structured audit narrative before trusting ML.

Tab 4 — Architecture selection  
- Prose comparing Transformers/CNNs vs PINNs vs hybrid digital-twin stacks.  
- Qualitative decision matrix (Approach vs Data need, Interpretability, Regulatory story, Fit for C-MAPSS).  
- Mermaid flowchart: fleet time series → data budget → sequence model vs PINN path → hybrid monitoring.  
- Second Mermaid diagram: why PINN can be “safer” than a pure black box for aerospace (staged assurance narrative).

Tab 5 — Deep learning lab (illustrative, not trained on user data)  
- Synthetic loss curves for CNN vs Transformer vs PINN-style model families.  
- Radar comparison of qualitative strengths across regimes, attention, data efficiency, interpretability, edge deploy, physics consistency.  
- Stacked bar of illustrative loss-term emphasis.  
- Three-panel intuition figure: CNN local window, Transformer attention mass, PINN wear channel.  
- Mermaid architecture flow: 1D CNN path, Transformer path, PINN path converging to C-MAPSS FD001–FD004 context.  
- Optional expandable implementation checklist.

Tab 6 — Compute & deployment  
- Qualitative discussion of GEMM dominance, KV cache scaling with history length, INT8/INT4 quantization.  
- Simple interactive “FLOP sanity check”: parameters × tokens heuristic.  
- Cost calculator: fleet size, inferences per aircraft per day, API $/1000 inferences vs self-hosted fixed + marginal cost—framed as a communication aid, not a full TCO model.
